# Implementation Plan: Merge Mesh And Material Renderer

**Branch**: `merge-mesh-material-renderer` | **Date**: 2026-05-22 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/merge-mesh-material-renderer/spec.md`

## Summary

Destructively merge `MaterialRenderer` into `MeshRenderer` so renderable mesh objects use `MeshFilter + MeshRenderer` only. Move material slots, material reference/path resolution, and user matrix state into `MeshRenderer`; update rendering, serialization/import/editor workflows to consume the merged component; remove the standalone `MaterialRenderer` component from production code and generated reflection.

## Technical Context

**Language/Version**: C++20 runtime/editor code; C#/.NET 8 MetaParser generator already present  
**Primary Dependencies**: Nullus component system, reflection runtime, MetaParser generated registration, object graph serialization, render scene cache, editor asset workflows  
**Storage**: Object graph/prefab serialized component records and material asset references  
**Testing**: GoogleTest via `NullusUnitTests`; `ReflectionTest` smoke validation; source-contract grep where useful  
**Target Platform**: Primary local validation on Windows; rendering behavior remains backend-agnostic  
**Project Type**: Native engine/runtime/editor component model refactor  
**Performance Goals**: Preserve render-scene material resolution cost; avoid additional per-frame component lookups compared with current render scene synchronization  
**Constraints**: Do not hand-edit `Runtime/*/Gen/` or `Project/*/Gen/`; no silent old-asset compatibility loader; avoid reverting unrelated dirty workspace changes; keep `MeshFilter` separate  
**Scale/Scope**: Runtime components/rendering/serialization/assets/editor/tests for the `MeshRenderer` and `MaterialRenderer` merge

## Phase 0: Research

### Decision: Destructive merge into `MeshRenderer`

**Rationale**: The user explicitly selected option C. A destructive merge removes the duplicated component model instead of preserving a deprecated `MaterialRenderer` path.

**Alternatives considered**:

- Compatibility migration loader. Rejected because the selected destructive path should expose stale assets/tests instead of silently accepting old schema.
- Soft deprecation with both components alive. Rejected because it preserves the dual-component model.

### Decision: Move material state and API rather than adding a nested helper component

**Rationale**: Existing render/editor code expects component-owned material APIs. Moving the state/API into `MeshRenderer` provides the smallest conceptual model and keeps reflection fields directly on the component.

**Alternatives considered**:

- Extract `MaterialSlotState` service class first. Rejected for the initial merge because it adds an abstraction before duplication is proven after relocation. A private helper can be introduced later if `MeshRenderer` becomes too large.

### Decision: Remove `MaterialRenderer` from reflection/build after call sites move

**Rationale**: Removing the type too early will break broad compilation. The safe order is to first add material capability to `MeshRenderer`, then update call sites/tests, then remove the old files from build and generated reflection flow.

**Alternatives considered**:

- Delete `MaterialRenderer` first and repair compile failures. Rejected because it creates noisy breakage across unrelated surfaces and makes generated-code boundaries harder to audit.

### Decision: Keep material serialized field names on `MeshRenderer`

**Rationale**: New object graphs need a direct place to store material references. Reusing `materials` and `userMatrixValues` on `MeshRenderer` keeps field semantics recognizable and makes tests straightforward.

**Alternatives considered**:

- Rename fields to `meshMaterials` or `rendererMaterials`. Rejected because it creates unnecessary schema churn inside the already destructive type move.

## Phase 1: Design

### Data Model

- **MeshRenderer**
  - Existing fields: `frustumBehaviour`, `customBoundingSphere`
  - Moved material fields: serialized material references, resolved material pointers, material paths, failed material paths, material display names
  - Moved user data: user matrix and reflected float-array bridge
  - Methods: frustum accessors plus material assignment, material resolution, material path/reference helpers, user matrix accessors

- **RenderPrimitive**
  - Existing owner/mesh renderer/mesh/cached command state remains
  - Removes separate `MaterialRenderer*`
  - Resolves materials from `MeshRenderer`

- **Model Prefab Mesh Node**
  - Components: transform, mesh filter, mesh renderer
  - Mesh renderer properties include frustum behavior and material references

### Contracts

No external API contract is added. Internal source/reflection contracts:

- `MeshRenderer` is the only renderable mesh component type.
- `MaterialRenderer` is not a production component type.
- Material fields on object graph records belong to `NLS::Engine::Components::MeshRenderer`.
- New assets use `MeshFilter + MeshRenderer`; old `MaterialRenderer` records are unsupported.
- Generated reflection output for `MeshRenderer` includes frustum and material fields.
