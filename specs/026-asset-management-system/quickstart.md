# Quickstart: Asset Management System

## Validate P1 Source Asset Foundation

1. Create a temporary asset root with `Models/Hero.gltf`, `Models/Hero.bin`, `Textures/Hero.png`, `Prefabs/Lamp.prefab`, and `Models/Legacy.fbx.meta`.
2. Run the source asset scan.
3. Verify `.meta` files exist for editable source assets and contain valid `GUID`, `IMPORTER_ID`, `IMPORTER_VERSION`, and `ASSET_TYPE` fields.
4. Verify `.gltf`, `.glb`, `.fbx`, and `.obj` classify as `ModelScene`; `.prefab` classifies as `Prefab`; textures classify as `Texture`.
5. Run the scan again.
6. Verify GUID values remain unchanged.
7. Rename `Models/Hero.gltf` and `Models/Hero.gltf.meta` together.
8. Run the scan again and verify lookup by GUID returns the new path.
9. Add `Models/Sponza.gltf` and `Models/Sponza.fbx` in the same folder.
10. Run the scan and verify both assets receive distinct `.meta` GUIDs, distinct ArtifactDB source records, and distinct extensionless content-addressed payload files under `Library/Artifacts/<first-two-hash-chars>/<hash>`.

## Validate External Model Import

1. Import a glTF 2.0 fixture with mesh, PBR material, external texture, skeleton, animation, and morph target.
2. Verify the external `.gltf`, `.bin`, and image dependencies are copied into project assets or recorded as explicit diagnostics if the user chooses not to copy them.
3. Verify generated Nullus assets include model, mesh, material, texture, skeleton, skin, animation clip, morph target, and generated prefab artifacts.
4. Import representative FBX and OBJ fixtures.
5. Verify FBX uses the common `ImportedScene` representation and records parser-exposed data or unsupported-channel diagnostics.
6. Verify OBJ generates static mesh/material/texture/model/prefab artifacts and records diagnostics for unsupported skeleton, skin, animation, and morph features.
7. Import the same fixture twice and verify sub-asset IDs remain deterministic.
8. Import same-stem model sources such as `Sponza.gltf` and `Sponza.fbx` and verify their artifacts do not overwrite each other even if both expose `prefab:Sponza`.
9. In the Asset Browser, verify the source model file is the visible draggable entry and no extra virtual `Sponza.prefab` child/file is shown for the generated model prefab.

## Validate Prefab Pipeline

1. Create a GameObject hierarchy with `TransformComponent`, `MeshFilter`, `MeshRenderer`, and at least one `$asset` reference.
2. Save it as `Assets/Prefabs/Test.prefab`.
3. Verify the source file is a `Nullus.ObjectGraph.Prefab` document with stable object IDs and a `.meta` GUID.
4. Import the prefab and verify a runtime prefab artifact is produced.
5. Instantiate the prefab into a scene and verify source-to-instance object ID mapping exists.
6. Add a local override, save the scene, reload it, and verify the override remains attached to the correct source object ID.
7. Create a prefab variant with a base prefab reference and verify base chain validation rejects cycles and missing bases.
8. Import a model fixture and verify its generated prefab is read-only by default and can be used as the base of an editable prefab variant.

## Validate Incremental Refresh

1. Modify one texture dependency used by a model import.
2. Refresh the asset database.
3. Verify only the dependent texture, material, model, generated prefab, and scenes/prefabs that reference them are stale.
4. Modify a base prefab.
5. Verify dependent variants, scenes, and build manifests are stale while unrelated assets remain up-to-date.
6. Trigger reimport and verify failed replacement imports keep the previous successful artifact resolvable.

## Validate Runtime Build Manifest

1. Build a runtime manifest from a scene referencing an imported model prefab and a user-authored prefab variant.
2. Verify the manifest includes every reachable runtime artifact exactly once.
3. Verify source-only files such as `.gltf`, `.glb`, `.fbx`, `.obj`, `.meta`, and editor cache databases are excluded unless explicitly marked for raw packaging.
4. Run the game from the build output directory.
5. Verify runtime resolves scene, prefab, mesh, material, texture, skeleton, animation, and morph artifact references through the manifest.

## Validate Editor Compatibility Fixtures

1. Create fixture assets covering model, texture, material, regular prefab, model prefab, prefab variant, nested prefab, missing prefab, and pack-style metadata.
2. Run editor-compatible AssetDatabase tests for path/GUID lookup, create/delete/copy/move/rename, folder operations, import/refresh batching, search filters, labels, dependency queries, main asset lookup, sub-asset lookup, and asset pack metadata.
3. Run editor-compatible importer tests for `GetAtPath`, serialized setting mutation, dirty state, `SaveAndReimport`, external object remaps, user data, target platform settings, postprocessors, scripted importers, and diagnostics.
4. Run editor-compatible PrefabUtility tests for prefab type/status, save-as-prefab, save-and-connect, prefab contents editing, Prefab Stage, variants, nested prefabs, missing prefabs, granular apply/revert, unpack, and unpack completely.
5. Verify every supported, diagnostic-only, deferred, and out-of-scope behavior is reflected in `contracts/editor-compatibility-matrix.md`.

## Validate Editor Drag/Drop Workflow

1. Drag a regular prefab asset from the Asset Browser onto the Hierarchy root and verify a connected prefab instance is created.
2. Drag a model source asset or prefab asset from the Asset Browser onto an existing Hierarchy object and verify it is parented under that object while remaining connected to the source prefab or generated model prefab artifact.
3. Drag a material asset onto a renderer in the Hierarchy and verify the target material slot stores a GUID-backed material reference and the scene is marked dirty.
4. Drag a texture asset onto a renderer and verify the editor creates or updates a material according to policy, then assigns that material to the renderer.
5. Drag an ordinary Hierarchy hierarchy into an Asset Browser folder and verify a `.prefab` source asset and `.meta` GUID are created.
6. Drag a connected prefab or generated model prefab instance into an Asset Browser folder and verify the editor creates a variant or unpacked copy according to the selected command.
7. Attempt invalid drops, such as material onto a non-renderer or prefab into a read-only package folder, and verify diagnostics are reported without mutating assets or scenes.
8. For model and prefab assets, verify Asset Browser publishes the imported asset drag payload. Scene View and Hierarchy must consume that payload before considering the legacy raw file payload.
9. Drop a model asset whose generated prefab artifact is missing or still importing. The drop must return pending or diagnostics, must not create scene objects, and must not synchronously import or write artifacts on the UI thread.
10. Drop a previously imported large model asset. The drop must create the connected generated prefab instance from committed artifacts, while mesh/material realization remains deferred through the editor resource-resolution queue.
11. Start the editor with cold `.gltf`/`.glb`/`.fbx`/`.obj` and `.prefab` files already under project `Assets`; verify Project Browser startup schedules background preimport and later drags publish warm imported handles instead of invoking import on drop.
12. Copy or move a model/prefab into project `Assets` while the editor is open; verify the file watcher reports the changed path, background preimport runs for that asset, and unrelated model/prefab assets are not force-reimported.
13. Verify a `.prefab` imported by background preimport stays warm after a fresh `AssetDatabaseFacade::Refresh()` because its manifest and runtime prefab payload are persisted under `Library`.
14. For existing projects with older imported model artifacts, verify `IMPORTER_VERSION=3` on model `.meta` files causes stale `.nmesh`, `.nmodel`, and `.mat` artifacts to be regenerated before warm drag/drop is allowed.

## Validate Connected Prefab Editing Workflow

1. Drag a `.prefab`, `.gltf`, `.glb`, `.fbx`, or `.obj` source asset from the Asset Browser into the Hierarchy. The created root must be a connected prefab instance with a stable prefab asset id, sub-asset key, source-to-instance mapping, and selected instance root.
2. Verify the Hierarchy text color distinguishes connected prefab roots, prefab children, local overrides, missing prefab assets, and generated read-only model prefab instances.
3. Double-click a `.prefab` source asset in the Asset Browser. The editor should open an isolated prefab stage, rebuild the Hierarchy against that stage scene, and focus the Scene View on the staged contents.
4. Edit staged root properties, run the normal Save command, and verify the prefab source text is rewritten, the stage becomes clean, and every connected open-scene instance refreshes from the saved prefab while preserving local overrides such as name, tag, and active state.
5. Close or discard the active prefab stage and verify the Hierarchy returns to the current scene without mutating scene instances beyond saved prefab propagation.
6. Treat structural stage edits, such as adding/removing/reordering child GameObjects and components, as Phase 23 work until `RefreshConnectedInstanceFromPrefab` rebuilds connected instance structure while preserving local overrides.

## Validate Async Import Progress

1. Start a batch import with a large glTF/GLB/FBX fixture and several texture dependencies.
2. Verify the editor command loop remains responsive while dependency copy, parse, conversion, artifact write, postprocess, and commit phases run.
3. Verify progress events include batch count, current asset, phase name, percent, diagnostics, cancellation state, and terminal result.
4. Cancel an import during dependency copy or artifact staging and verify staged files are discarded while the previous successful artifact remains active.
5. Force an import failure after artifact staging and verify progress reports failure only after rollback preserves the previous committed manifest.

## Validate Imported Material Visual Mapping

1. Import a glTF PBR fixture with base color, metallic-roughness, normal, occlusion, emissive, alpha blend/cutoff, UV transform, and sampler state.
2. Verify generated material artifacts store expected texture slots, factors, color-space policy, alpha mode, normal scale, emissive factor, and sampler data.
3. Import an FBX material fixture and verify parser-exposed diffuse/base color, specular/glossiness or roughness, normal/bump, emissive, opacity, texture paths, and UV channels are mapped or diagnosed.
4. Import an OBJ+MTL fixture and verify diffuse, specular, shininess, opacity/dissolve, normal/bump, and illumination hints are mapped or diagnosed.
5. Instantiate the generated model prefab and verify it references converted material artifacts rather than source material records.
6. Verify editor preview, scene viewport, runtime manifest, and build output all resolve the same converted material artifacts.
7. Run `AssetMaterialViewportTests` for generated glTF, FBX, and OBJ prefab fixtures and verify the editor preview or scene viewport binds the converted material artifacts.
8. Capture RenderDoc evidence or run deterministic renderer screenshot validation for the same fixtures, then record the artifact paths, fixture names, backend, and expected channel coverage here before claiming visual correctness.
9. Import a glTF/GLB fixture with two meshes whose nodes reference meshes out of source order, plus UV0 accessors. Verify `mesh:<sourceKey>` artifacts preserve their original mesh positions and UVs instead of using parser traversal order.
10. Import FBX/OBJ fixtures with independent metallic and roughness textures. Verify generated `.mat` files contain `u_MetallicMap` and `u_RoughnessMap` independently, falling back to packed metallic-roughness only when independent slots are absent.
11. Verify generated `.mat` sampler uniforms resolve to committed texture artifact payloads under `Library/Artifacts/<first-two-hash-chars>/<hash>`, and `TextureLoader` can decode those payloads into runtime `Texture2D` resources using ArtifactDB metadata instead of filename extensions.

## Current Implementation Notes

- Source identity, `.meta` preservation, duplicate GUID diagnostics, and `.prefab` classification live under `Runtime/Core/Assets`.
- glTF import covers glTF 2.0 JSON contract metadata plus GLB container parsing for JSON/BIN chunks, bufferViews, accessors, embedded image bufferViews, primitive vertex stream accessors, and index accessors.
- `Runtime/Core/Assets/ArtifactWriter` stages model-derived payloads before commit, writes manifest entries with loader IDs/content hashes, rejects path escapes, and rolls back committed-file replacements if a multi-payload commit fails.
- Model import manifests now record source stamp dependencies for glTF external buffers/images and OBJ `.mtl`/texture files under project `Assets`; the preimport scheduler, Asset Browser drag payloads, and imported-handle drop fast path all treat changed or missing dependency stamps as cold/pending rather than reusing stale artifacts.
- Model imports now write a real `model.nmodel` package index beside mesh/material artifact payloads under the content-addressed `Library/Artifacts/<first-two-hash-chars>/<hash>` store. `ModelManager` can resolve a requested mesh artifact through that package, register the main model cache entry, and register sibling mesh aliases so large imported models do not fall back to independent per-mesh source parsing or cache misses.
- Blocking startup preimport now warms the runtime-visible cache for existing scene-droppable model assets before the normal editor UI opens: it loads each imported primary `.nmodel` package with project-relative `Library/...` keys, registers mesh aliases through `ModelManager`, and prewarms generated `.mat` material artifacts through `MaterialManager::PrewarmArtifact` only when material, shader, and texture manager services are available. This mirrors the reference-editor drag path: once the UI is visible, an existing imported model drop should instantiate from committed artifacts and cached resource handles; only newly added or changed assets run post-open background preimport.
- Generated model renderer resource resolution now schedules a main model package task before mesh alias tasks, checks the `ModelManager` cache before starting any per-mesh `.nmesh` artifact load, applies a visible fallback material immediately, preserves real material path hints, and binds real materials only when `MaterialManager::GetResource(path, false)` already has them cached. Model package prewarm must actually return a loaded cache entry before mesh alias tasks proceed; a missing or invalid `.nmodel` now fails the renderer resource job with a diagnostic instead of silently degrading into many cold per-mesh reads or retrying forever.
- `ModelManager` now canonicalizes project `Library/...` model and mesh artifact requests before loading packages. Relative startup prewarm and later absolute artifact requests reuse the same cached `Model*` entries, preventing duplicate `.nmodel` reads, cache-key inflation, and replacement of pointers that may already be bound to scene renderers.
- FBX and OBJ import use the existing parser adapter path and convert parser-exposed data into `ImportedScene`; unsupported parser channels are reported as diagnostics. The external model importer now reads FBX/OBJ source through Assimp once per import, reusing the same parsed mesh/material data for both fallback scene records and `.nmesh` artifact payloads. Parsers that implement the optional imported-scene data provider can expose OBJ MTL texture dependencies and FBX hierarchy/material/texture/skeleton/skin/animation/morph records without changing the legacy mesh-only parser path.
- FBX/OBJ Assimp detailed import now carries parser-exposed material channels into converted `.mat` artifacts: diffuse/base color, normal/bump, opacity, shininess, roughness, metallic, occlusion, emissive, specular, double-sided flags, and texture URIs. Independent metallic and roughness maps are serialized as separate material uniforms, with packed metallic-roughness used only as fallback.
- Imported texture sub-assets now write extensionless content-addressed artifacts with source or embedded encoded image bytes after `PAYLOAD_BEGIN`; generated `.mat` sampler uniforms reference those committed artifact paths instead of source image files. `TextureLoader` unwraps imported texture payloads before stb image decode, so `TextureManager` can bind imported texture artifacts directly during warm model material realization.
- Startup material prewarm can intentionally leave declared sampler uniforms unloaded; generated model resource resolution repairs those declared texture bindings before binding real materials, and the material loader only logs texture load warnings when texture loading was requested and failed.
- Native mesh artifacts are keyed to source-local mesh identities instead of parser traversal vector indices. FBX/OBJ parser meshes carry `sourceKey`/`sourceMeshIndex`, and glTF/GLB `.nmesh` payloads are decoded directly from source accessors/buffers by original mesh index while preserving positions, indices, UV0, normals, tangents, and multi-primitive merges. Model importer version `3` invalidates older placeholder or white-material artifacts on refresh.
- Model imports can generate read-only prefab artifacts through `Runtime/Engine/Assets/ModelPrefabBuilder`, mapping imported nodes to `GameObject`, `TransformComponent`, `MeshFilter`, `MeshRenderer`, and `$asset` references.
- Skeleton, skin, animation, and morph records are preserved as sub-assets. Generated static model prefabs emit explicit `runtime-skinning-component-missing` and `runtime-morph-component-missing` diagnostics until reflected runtime playback components exist for skinning, animation clip evaluation, and morph targets.
- Runtime prefab helpers can instantiate prefab artifacts with stable source-to-instance object maps, validate base-chain missing/cycle cases, and report unresolved `$asset` references.
- `Project/Editor/Assets/PrefabEditorWorkflow` now provides the initial editor command layer for create-prefab-from-selection and prefab instantiation, including multi-root rejection, generated source text, artifact import, scene attachment, and local patch storage.
- Incremental reimport is represented by `AssetVersion`, `AssetDependencyGraph`, `ImportDiagnosticList`, and `AssetResolver`; failed imports retain the previous committed manifest.
- `Project/Editor/Assets/EditorAssetDatabase` exposes generated prefab read-only state, variant/unpack policy, and a refresh scheduler bridge for `AssetFileWatcher` events.
- `Project/Editor/Core/AssetFileWatcher` now exposes readiness after platform watcher registration. On Windows, `Start()` uses an overlapped `ReadDirectoryChangesW` request before reporting ready, and `AssetFileWatcherTests.StartReportsReadyBeforeImmediateCreateEvents` validates immediate create events after startup.
- Runtime builds use `RuntimeAssetManifest` and `RuntimeAssetDatabase` to resolve GUID/sub-asset references. Legacy path-based lookups can be bridged during migration, but new package resolution is manifest-first.
- editor compatibility scope is tracked in `contracts/editor-compatibility-matrix.md`; intentional differences must be recorded there before completion is claimed.
- Asset Browser/Hierarchy drag-drop and async import progress have automated coverage through `EditorAssetDragDropTests` and `AssetImportProgressTests`.
- Imported material conversion and preview/scene binding have automated coverage through `AssetMaterialConversionTests`, `AssetPrefabPipelineTests`, and `AssetMaterialViewportTests`. Pixel-level visual evidence is recorded through the deterministic screenshot report listed above.
