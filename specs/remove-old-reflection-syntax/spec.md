# Feature Specification: Remove Old Reflection Syntax

**Feature Branch**: `remove-old-reflection-syntax`
**Created**: 2026-05-07
**Status**: Draft
**Input**: User description: "把所有的旧的反射写法都删掉；外部反射不要采用代码生成的方式，包装反射宏，手动注册到反射就行"

## User Scenarios & Testing

### User Story 1 - AST-Only Generated Reflection (Priority: P1)

Engine developers can mark reflected declarations with concise macros that include the C++ declaration keyword, then rely on MetaParser to read only the AST attribute data.

**Why this priority**: This removes duplicate call-site syntax and keeps generated reflection tied to valid C++ declarations.

**Independent Test**: Build the reflection test target and inspect generated files for `CLASS(Foo)`, `STRUCT(Foo)`, `ENUM(Foo)`, `PROPERTY()`, and `FUNCTION()` declarations without old explicit member directives.

**Acceptance Scenarios**:

1. **Given** a header with `CLASS(Foo)` and `FUNCTION()` on an overloaded const method, **When** MetaParser runs, **Then** the generated method binding uses an unambiguous function pointer.
2. **Given** a reflected type deriving from `NLS::meta::Object`, **When** generated headers are emitted, **Then** `GENERATED_BODY()` supplies the object bridge methods.

### User Story 2 - Manual External Reflection (Priority: P1)

Developers can register external reflection bindings manually through wrapper macros instead of feeding external reflection syntax to MetaParser.

**Why this priority**: External reflection should not be generated by MetaParser and should remain explicit runtime registration code.

**Independent Test**: Build and run reflection tests that require `Vector3`, serialization DTO, and private external sample bindings while confirming no generated external reflection files are referenced.

**Acceptance Scenarios**:

1. **Given** an external reflection header, **When** the runtime reflection database initializes, **Then** its static registrar manually declares and defines the reflected type.
2. **Given** old `REFLECT_EXTERNAL` syntax is searched in runtime headers, **When** the search completes, **Then** no old call-site usage remains.

### Edge Cases

- Overloaded methods must be emitted with `static_cast` signatures so the old `FUNCTION(pointer = ...)` spelling is unnecessary.
- Pointer-returning getter/setter pairs must not become reflected properties unless there is a supported value-type adapter pair.
- External manual registration must support module keys, dependencies, fields, instance methods, static methods, and enum-less class/struct types.

## Requirements

### Functional Requirements

- **FR-001**: Generated reflection call sites MUST use `CLASS(typeDecl)`, `STRUCT(typeDecl)`, and `ENUM(typeDecl)` without repeating the C++ keyword after the macro.
- **FR-002**: MetaParser MUST use the CppAst AST path for generated reflection and MUST NOT parse headers through the old text parser route.
- **FR-003**: Explicit generated-reflection member directives such as `PROPERTY(name = ...)` and `FUNCTION(pointer = ...)` MUST be removed from runtime call sites.
- **FR-004**: External reflection bindings MUST not be generated by MetaParser.
- **FR-005**: External reflection wrapper macros MUST manually register reflected types into the reflection database through runtime code.
- **FR-006**: Existing reflected runtime behavior required by unit and reflection tests MUST remain available except intentionally removed fields.

### Key Entities

- **Generated Reflection Type**: A class, struct, or enum emitted by MetaParser from AST attributes.
- **External Reflection Binding**: Runtime registration code for a type whose reflection data is not generated by MetaParser.
- **Object Bridge**: Generated `GetType` and `Clone` members for reflected types derived from `NLS::meta::Object`.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Searches for old external-reflection call-site macros in runtime headers return no usage outside macro definitions.
- **SC-002**: `ReflectionTest` builds and runs with zero failures.
- **SC-003**: `NullusUnitTests` builds, and the reflection-related unit tests pass.
- **SC-004**: Generated module sources no longer include external reflection generated files.

## Assumptions

- Direct `MetaExternal` type-name helper usage can remain only as a wrapper implementation detail if needed, not as a user-facing external-reflection binding syntax.
- Generated files under `Runtime/*/Gen/` remain generated output and are updated only by the build flow.
