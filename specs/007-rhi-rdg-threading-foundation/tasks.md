Progress note (2026-04-20): the first five Phase 4B slices are landed and verified. `ABaseRenderer`, `CompositeRenderer`, `BaseSceneRenderer`, `ForwardSceneRenderer`, and `DeferredSceneRenderer` can now publish prepared render-scene builders that are executed during the render-scene completion stage, instead of publishing authoritative `RenderScenePackage` objects from the calling thread for the normal threaded path. In the current steady-state threaded path, `ForwardSceneRenderer` and `DeferredSceneRenderer` no longer retain a calling-thread `pendingRenderScenePackage`; their builders now capture snapshot-owned data and prepared light-grid dispatch inputs by value, avoiding renderer-lifetime capture. Generic `CompositeRenderer`-based snapshot renderers now also publish prepared builders instead of raw snapshots, shrinking `DriverWorkerFallback` usage to manual snapshot compatibility paths plus drain/shutdown behavior. `Driver` now tracks prepared-builder publication origin explicitly, preserves the old backpressure/wait semantics for builder publication, treats missing prepared payload on the steady-state prepared path as an explicit `PreparedPayloadMissing` dropped package instead of silently synthesizing a driver-built scene package, rejects direct snapshot/prepared-package compatibility publication for all requested runtime backends except the `NONE` debug/test harness, and splits `ResolveThreadedRenderSceneCompletion(...)` into explicit compatibility-vs-prepared resolution helpers so drain/worker paths no longer share a monolithic fallback builder branch. Resolver-side compatibility scene-package construction is now also hard-gated to `requested backend = NONE` or `SynchronousDrainFallback`; even if a non-`NONE` runtime path bypasses the publish gate and injects a snapshot-compatibility slot directly into the lifecycle, `Driver` now retires it as `CompatibilityRejected` with an empty package instead of synthesizing driver-owned scene work. For the non-threaded graph path, `LightGridPrepass` preparation is no longer appended into `pendingRenderScenePackage.computeDispatchInputs` during `BeginFrame()`. `ForwardSceneRenderer` and `DeferredSceneRenderer` now register prepared light-grid work as graph-owned callback passes before scene execution, and the obsolete base-renderer package-injection helper has been removed. For the threaded ordered-submit path, `Driver` no longer records `computeDispatchInputs` in a separate pre-pass command-buffer branch; the prepared light-grid dispatches are now folded into the ordered work-unit skeleton as explicit compute work units. Those compute work units are no longer prepended by renderer-specific package surgery after `CompileAndApplyThreadedRenderSceneExecution(...)`; `FrameGraphExecutionPlan` now accepts additional ordered work units directly and applies the final threaded package ordering. In the latest slice, `LightGridPrepass` also exposes compiled-pass metadata plus graph/threaded pass builders, and `ForwardSceneRenderer` / `DeferredSceneRenderer` now compile a combined `LightGrid + scene-pass` sequence so `Injection` / `Compact` become first-class entries in the compiled pass list instead of renderer-local threaded payload appended outside the compiled graph result. `T036`, `T037`, and `T039` remain open until steady-state DX12/Vulkan paths stop treating calling-thread package publication as authoritative, `RenderScenePreparing` becomes the fully authoritative render-scene/RDG build stage, and the threaded path stops depending on graph-external light-grid dispatch injection.
