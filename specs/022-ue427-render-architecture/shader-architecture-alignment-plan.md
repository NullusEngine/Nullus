# UE4.27 Shader Architecture Alignment Plan

> Scope: align Nullus shader definition, compilation ownership, shader maps, parameter binding, and RDG-style dispatch/draw binding with UE4.27 concepts without directly copying UE source.

## Current Status

Nullus now has a lightweight `ShaderParameterStruct` path and some renderer-owned parameter definitions for `Standard`, `Lambert`, `StandardPBR`, `DeferredLighting`, and LightGrid passes. This is not yet UE4.27 source-level shader architecture parity.

The main gap is that Nullus still uses asset/path-driven `ShaderLoader` and `Material` binding creation, while UE4.27 uses statically registered shader types (`FShaderType`, `FGlobalShader`, `FMaterialShader`), shader maps, permutation domains, root parameter metadata, and RDG pass parameter binding.

## Target Architecture

Nullus should have these UE-aligned layers:

1. **Shader Type Registry**
   - Static renderer-owned shader definitions replace filename switches.
   - Each shader type owns source path, entry point, stage, permutation domain, compile rules, and root parameter struct metadata.

2. **Shader Classes**
   - `GlobalShader` becomes a base class for engine/global shaders.
   - `MaterialShader` becomes the base for material-bound shaders.
   - Concrete shaders such as `StandardPS`, `LambertPS`, `DeferredLightingPS`, `LightGridInjectionCS` are declared in C++.

3. **Shader Parameter Metadata**
   - Parameter structs are no longer loose runtime vectors only.
   - Each shader class exposes root parameters analogous to `FShader::GetRootParametersMetadata()`.
   - Layout creation, binding set creation, validation, and shader compiler binding contracts use that metadata.

4. **Shader Maps**
   - Global shaders are resolved through a global shader map.
   - Material shaders are resolved through a material shader map keyed by material domain, feature flags, permutation, and backend.
   - `Material` should request shader map entries instead of owning graphics pipeline binding layout generation directly.

5. **RDG-Style Pass Binding**
   - Frame graph passes allocate pass parameters with typed parameter structs.
   - Pass execution binds shaders via shader refs and calls a central `SetShaderParameters` equivalent.
   - LightGrid and deferred lighting should be the first fully migrated passes.

6. **Reflection Fallback**
   - HLSL reflection remains only as a validation/fallback path for external or legacy shaders.
   - Engine shaders should fail tests if they rely on reflection-only layout creation.

## Definition Of Done

Shader architecture can be called UE4.27-aligned only when all items are true:

- Engine shaders are declared as C++ shader classes, not filename cases in `ShaderLoader`.
- Every migrated shader has root parameter metadata owned by the shader class.
- Global and material shader maps resolve compiled shader instances.
- Renderer passes use shader refs and typed pass parameters.
- LightGrid and DeferredLighting use RDG-style pass parameter binding.
- Standard/Lambert/StandardPBR/DeferredGBuffer use material shader classes.
- Reflection-only layout creation is limited to external/custom shaders.
- DX12 runtime evidence exists with no descriptor table binding spam.
- Documentation records remaining non-parity items, if any.

## Recommended Execution Order

1. Phase 0 tests.
2. Phase 1 registry.
3. Phase 2 metadata.
4. Phase 3 global shader classes for LightGrid and DeferredLighting.
5. Phase 4 shader maps.
6. Phase 5 material shader classes.
7. Phase 6 RDG-style pass parameter binding.
8. Phase 7 legacy removal.
9. Phase 8 runtime evidence.

This order keeps the engine runnable after each step and avoids a single risky rewrite.
