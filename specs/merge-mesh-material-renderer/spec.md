# Feature Specification: Merge Mesh And Material Renderer

**Feature Branch**: `merge-mesh-material-renderer`  
**Created**: 2026-05-22  
**Status**: Draft  
**Input**: User description: "合并MaterialRenderer MeshRenderer" with explicit choice "C" for destructive merge

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Single Mesh Rendering Component (Priority: P1)

Engine and editor users need one component to represent renderable mesh state, so a mesh object no longer requires both `MeshRenderer` and `MaterialRenderer` to render with assigned materials.

**Why this priority**: This is the core requested behavior. It simplifies object composition and aligns Nullus mesh rendering closer to the common `MeshFilter + MeshRenderer` model.

**Independent Test**: A focused render-scene test can create a `GameObject` with `MeshFilter` and `MeshRenderer`, assign materials through `MeshRenderer`, synchronize `RenderScene`, and verify visible draw commands use the assigned material.

**Acceptance Scenarios**:

1. **Given** a mesh object with `MeshFilter` and `MeshRenderer`, **When** a material is assigned through `MeshRenderer`, **Then** the render scene resolves that material without requiring a `MaterialRenderer` component.
2. **Given** a mesh object with `MeshFilter` and `MeshRenderer`, **When** no explicit material is valid, **Then** the render scene falls back to the configured default material exactly as the current material path does.
3. **Given** a mesh object with only `MeshFilter` and `MeshRenderer`, **When** the scene fast-access cache is rebuilt, **Then** the object appears once in the renderer fast-access list.

---

### User Story 2 - Destructive Removal Of MaterialRenderer (Priority: P1)

Maintainers need `MaterialRenderer` removed as an independent component type so runtime, editor, reflection, and tests stop preserving a duplicated rendering component model.

**Why this priority**: The user selected the destructive merge path. Keeping `MaterialRenderer` available or silently migrating old assets would preserve the old model and fail the requested simplification.

**Independent Test**: Reflection/component picker/source-contract tests can verify `MaterialRenderer` is no longer registered as a component, no model prefab generation emits `MaterialRenderer`, and production code does not call `GetComponent<MaterialRenderer>()`.

**Acceptance Scenarios**:

1. **Given** component reflection metadata, **When** the engine component types are queried, **Then** `MeshRenderer` exposes material fields and APIs while `MaterialRenderer` is absent.
2. **Given** a generated model prefab, **When** the object graph is inspected, **Then** mesh nodes contain one `MeshRenderer` record with both frustum and material properties, and no `MaterialRenderer` record.
3. **Given** production source code, **When** component usage is scanned, **Then** material assignment and material resolution are routed through `MeshRenderer`, not `MaterialRenderer`.

---

### User Story 3 - Editor And Importer Workflows Use MeshRenderer Materials (Priority: P2)

Editor workflows need material drops, asset previews, prefab opening, and imported model instantiation to operate against `MeshRenderer` material slots after the component merge.

**Why this priority**: Runtime rendering can be made correct first, but editor/importer workflows must follow so users can still author and inspect rendered objects.

**Independent Test**: Existing asset import, prefab, component picker, reflected property drawer, and editor action tests can be updated to assert material data lives on `MeshRenderer`.

**Acceptance Scenarios**:

1. **Given** a model asset import, **When** the prefab artifact is generated, **Then** imported material references are stored on each mesh node's `MeshRenderer`.
2. **Given** a material asset drag/drop target, **When** the target has a `MeshRenderer`, **Then** the material is assigned to the `MeshRenderer` material slots.
3. **Given** an asset preview scene, **When** the preview object is created, **Then** it uses `MeshFilter + MeshRenderer` only.

### Edge Cases

- Existing scene or prefab files that still contain `NLS::Engine::Components::MaterialRenderer` records are intentionally unsupported by this destructive merge and should fail or be updated to the new schema by tests/fixtures.
- Existing reflected fields named `materials` and `userMatrixValues` must remain available on `MeshRenderer` so new serialized objects can preserve material references and user matrix data.
- Generated files under `Runtime/*/Gen/` and `Project/*/Gen/` must not be hand-edited; reflection output must be regenerated by the normal MetaParser/build flow.
- Source files may already be dirty in the user workspace; implementation must avoid reverting unrelated changes.
- Rendering behavior must remain backend-agnostic: this change routes data differently but must not assume one graphics backend proves another backend.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `MeshRenderer` MUST own all material-slot state previously owned by `MaterialRenderer`, including resolved material pointers, serialized material references, material path hints, and failed material paths.
- **FR-002**: `MeshRenderer` MUST own the user matrix state and reflected `userMatrixValues` accessors previously owned by `MaterialRenderer`.
- **FR-003**: `MeshRenderer` MUST preserve its existing frustum behavior state, custom bounding sphere state, copy behavior, assignment behavior, and component menu identity.
- **FR-004**: `MeshRenderer` MUST expose material assignment and resolution APIs equivalent to the current `MaterialRenderer` public behavior needed by runtime/editor/tests.
- **FR-005**: `RenderScene` MUST resolve materials directly from the primitive's `MeshRenderer` and MUST NOT depend on a sibling `MaterialRenderer`.
- **FR-006**: `MeshFilter` mesh changes MUST trigger material-slot updates on the owning `MeshRenderer`.
- **FR-007**: Primitive creation, model prefab generation, asset import, prefab instantiation, and asset preview workflows MUST create mesh objects with `MeshFilter + MeshRenderer` only.
- **FR-008**: Model prefab object graphs MUST store material references on the `MeshRenderer` record and MUST NOT emit `MaterialRenderer` component records.
- **FR-009**: Editor material drag/drop and prefab resource binding workflows MUST target `MeshRenderer` material data.
- **FR-010**: The `MaterialRenderer` component type MUST be removed from production component registration, editor component search, runtime rendering code, and generated reflection outputs.
- **FR-011**: Tests and object graph fixtures MUST be updated to the new destructive schema rather than relying on old `MaterialRenderer` compatibility.
- **FR-012**: Production code MUST NOT preserve a compatibility loader that silently converts `MaterialRenderer` records into `MeshRenderer` data.
- **FR-013**: Generated reflection outputs MUST be produced only through the normal MetaParser/build flow.
- **FR-014**: Validation MUST include focused rendering/material-cache, object graph/prefab/import, reflection/MetaParser, and editor workflow tests.
- **FR-015**: Final review MUST include the repository plan-review gate before reporting implementation completion.

### Key Entities

- **MeshRenderer**: The single renderable mesh component. It combines frustum/culling state with material-slot state and user matrix data.
- **MeshFilter**: The component that continues to provide the mesh resource for a renderable object.
- **Material Slot State**: The material reference array, resolved material pointer cache, path hints, failed paths, and material slot display names needed to bind render resources.
- **Renderable Mesh Object**: A game object with `MeshFilter` and `MeshRenderer` that can produce render scene draw commands.
- **Model Prefab Record**: Serialized object-graph representation for imported mesh nodes, now containing one merged `MeshRenderer` component record.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: New or updated render-scene tests prove a `MeshFilter + MeshRenderer` object renders with assigned materials without adding `MaterialRenderer`.
- **SC-002**: Source and reflection tests report zero production `GetComponent<MaterialRenderer>()`, `AddComponent<MaterialRenderer>()`, or component-menu registration usage after the merge.
- **SC-003**: Generated model prefab tests show zero `MaterialRenderer` records and one material-bearing `MeshRenderer` record per mesh node.
- **SC-004**: Reflection tests show `MeshRenderer` exposes `frustumBehaviour`, `customBoundingSphere`, `materials`, and `userMatrixValues`, while `MaterialRenderer` is absent from expected component type coverage.
- **SC-005**: Focused unit tests covering render scene cache, object graph serialization, asset prefab/import, reflected property drawing, and editor component picking pass after the normal build regenerates reflection outputs.

## Assumptions

- The user explicitly selected destructive merge option C, so old serialized assets containing `MaterialRenderer` are not required to load successfully.
- The component remains named `MeshRenderer`; no new `Renderer` base class or new component name is introduced in this change.
- `MeshFilter` remains separate and continues to own mesh references.
- The public material API can keep current method names where practical to reduce call-site churn, but its owning type changes to `MeshRenderer`.
- Validation will be performed on the local Windows workspace unless the implementation phase identifies an available cross-platform runner.
